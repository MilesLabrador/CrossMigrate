Feature

Field-stat filtering — drop sparse/low-quality columns

The Filter node previously only filtered rows by a cell's value. It now supports a second condition scope: field, which filters out entire columns based on computed statistics — fill %, unique count, null count, inferred type, or field name.

Why: useful for data-cleaning before export — e.g. "drop columns with fill % below 90" or "drop columns with 0 unique values" — without needing a separate node type.

Each condition in the UI can now be toggled between Row and Field scope, with the operator list and value input adapting to the selected stat (numeric comparisons for fill %, an enum dropdown for type, text ops for name). The canvas node badges field-scope conditions in green so they're visually distinct from row filters at a glance.

Feature

Right-click context menu on canvas nodes

Right-clicking a node now opens a small dropdown with Duplicate and Delete. When multiple nodes are selected, the same right-click instead offers Duplicate N nodes, Group N nodes, and Delete N nodes, operating on the whole selection at once.

Why: requested directly — there was previously no way to remove or copy a node except dragging it to delete or rebuilding it from scratch.

  • Bug: right-click would open the menu then immediately close it on the same click, because a global "click outside closes the menu" listener was attached synchronously and caught the very event that opened it. Fixed by deferring listener attachment to the next animation frame.
  • Bug: right-clicking on a node that was part of a box-selected group fell through to the browser's native context menu instead of the app's. React Flow renders an invisible drag overlay (.react-flow__nodesselection-rect) on top of multi-selected nodes for group-dragging, and right-clicks land on that overlay, not the node — so onNodeContextMenu never fired. Fixed by also wiring up React Flow's onSelectionContextMenu, which is built for exactly this case.

Fix

Shift-drag box select wasn't selecting multiple nodes

Holding Shift and dragging on empty canvas is supposed to draw a marquee and select every node inside it. This was completely broken — the canvas always forced each node's visual "selected" state from a single selectedNodeId on every render, which silently overwrote whatever React Flow's own multi-select logic had just computed.

Why: requested directly, and needed as a foundation for the new bulk delete/duplicate/group context-menu actions above.

Fixed by letting React Flow own each node's selected flag directly (no re-derivation), and syncing the app's single-selection concept (used to open the config panel) via React Flow's onSelectionChange/onSelectionStart/onSelectionEnd hooks, buffering changes during an active drag so the panel doesn't open/close mid-drag.

Feature

Node grouping with notes

Selecting 2+ nodes and choosing Group N nodes from the context menu wraps them in a new Group node — a resizable, dashed-border container with a draggable title bar. Group nodes carry a name and a free-text notes field, useful for annotating why a cluster of nodes exists.

Why: requested directly, modeled on the "frame" concept from design tools like Figma — a way to visually organize and document a pipeline without affecting its execution.

Clicking a group's title bar opens the same right-hand config panel used by every other node, showing the editable name and notes textarea. Deleting a group ungroups its children (converting their position back to absolute canvas coordinates) rather than deleting them — grouping is purely organizational, never destructive. Group nodes are excluded from the payload sent to /api/run-pipeline since they have no data role.

Fix

Node selection & click-to-delete — a three-round bug chase

This was the most involved fix of the session, and worth documenting in full because the first two attempts each fixed one symptom while introducing another.

  • Round 1 — "single click doesn't select the node": diagnosed (incorrectly) as NodeShell's stopPropagation() fully blocking React Flow's native selection. Fix attempt: made the selectNode store action also write selected: true/false directly onto node objects.
  • Regression from round 1: clicking back and forth between two edge-connected nodes (e.g. CSV Input → Preview Columns) crashed the app to a black screen (React's "Maximum update depth exceeded", unmounting with no error boundary). The redundant, app-level selection write fought with React Flow's own internal selection/edge bookkeeping specifically when an edge was present.
  • Round 2 — revert: removed the direct node-mutation from selectNode, restoring stability. But this reintroduced round 1's original symptom for real: clicking a node now neither highlighted it nor let Backspace delete it (delete-by-key depends on React Flow's own node.selected, which was never being set).
  • Round 3 — actual root cause: NodeShell's outer click handler really was calling e.stopPropagation(), which does block React Flow's native click-to-select (wired on the parent node wrapper). The correct fix was simply to stop swallowing the event — let the click bubble so React Flow's own mechanism owns selected, and use the local click handler only to open the config panel.

Why this took three passes: synthetic test clicks (dispatched via JS, not real mouse events) don't reproduce browser click semantics exactly — React Flow's selection is actually driven by a native pointerdown listener from its drag library, not the React-level click event, so naive single-event tests gave misleading signal at each step. The final fix was verified with a realistic pointerdown→pointerup→click sequence across multiple connected nodes, confirming both visual selection and Backspace-delete work without the crash recurring.
